IntroduoIntroduo API Esta a documentao da API do REP iDClass. Para conferir nossa Coleo do Postman, com exemplos de todas as requisies apresentadas aqui, acesse este link. *** ATENO! ***

Algumas chamadas da API foram adaptadas para compatibilidade com a Portaria 671 e agora recebem um parmetro pela query string chamado "mode". Se esse valor for 671, as chamadas consideram as mudanas da Portaria 671. Para qualquer outro valor ou na ausncia do parmetro, levado em considerao o funcionamento anterior do REP.

Esta API composta de Web Services RESTful. Cada comando uma chamada HTTPS POST com Content-Type: application/json e possui uma URL prpria. Seus parmetros e seus retornos so passados preferencialmente por JSON, exceto em meno contrria. Como exemplo, o comando login pode possuir a URL: https://192.168.0.129/login.fcgi e os seguintes parmetros: { login: 'admin', password: 'admin' } *** ATENO! ***

Seguindo as exigncias do INMETRO, o relgio comunica-se usando SSL (HTTPS). Como o relgio usa certificados auto-assinados, este fato pode gerar alguns problemas, como erros de comunicao inesperados. No final do documento h instrues de como evitar esse problema em algumas linguagens.

Todos os comandos requerem uma sesso vlida, exceto session_is_valid e login. O corpo da requisio deve possuir a codificao UTF-8. Caso os caracteres enviados no possuam nenhum caractere especial (ASCII), o corpo da requisio no precisa ser codificado.Por padro, a codificao utilizada por sistemas Windows a Windows-1252 (tambm conhecida como CP-1252). Verificar como utilizar a codificao correta.
Valores Padro
A porta de comunicao 443. O usurio "admin". A senha "admin".
Executando os Exemplos
Os exemplos dessa documentao so escritos em JavaScript, utilizando a biblioteca jQuery. Para test-los, acesse o endereo IP do equipamento em um navegador web e use as ferramentas do desenvolvedor (developer tools) deste para realizar requisies. Todos os exemplos fornecidos podem ser verificados copiando seus cdigos e colando no console das ferramentas de desenvolvimento. Devido ao fato das requisies virem de uma fonte diferente do relgio, a maioria dos browsers modernos bloqueia a requisio (CORS). No Google Chrome, isso pode ser ignorado executando-o com as flags --disable-web-security --ignore-certificate-errors

ChangelogChangelog da API iDClass Esta a relao de alteraes da API do REP iDClass para cada verso de firmware. Para obter a verso de firmware de seu REP, utilize a API get_about, cuja documentao se encontra em Sistema >Informaes do REP nesta mesma pgina. O parmetro versionFW conter o resultado em base decimal. Changelog API iDClass - verso 102 (ou 258 decimal):
  • [Facial] Novas APIs user_set_image, user_get_image e user_destroy_image para gerenciamento de fotos de usurios;
  • [Facial] Nova API user_import_image para gerenciamento de importao de fotos de usurios durante procedimento de backup;
  • [Facial] Novas APIs remote_enroll, facial_enroller_state e cancel_remote_enroll para gerenciamento de cadastro facial remoto;
  • [Facial] Novas APIs get_facial_module_config e set_facial_module_config para gerenciamento de configuraes faciais;
  • [Facial] Nova API get_about_facial para obter informaes do mdulo facial;
  • [Facial] Nova API set_facial_limit para realizao de upgrade para 10k faces do REP Facial;
  • [Facial] APIs add_users e update_users recebem novo parmetro geral "do_match" (booleano) e novos parmetros especficos de cada usurio "remove_faces" (booleano), "image" (em base 64) e "image_timestamp" (inteiro);
  • [Facial] API load_users retorna "face" (booleano) e "image_timestamp" (inteiro);
  • [Facial] API get_system_information retorna "number_of_faces" (string) em vez de "template_count" (inteiro);
  • API get_about retorna "isFacial" (booleano);
  • API get_system_network retorna "use_dhcp" (booleano) e "port" (inteiro);
  • API set_system_network recebe "use_dhcp" (booleano que, sempre que no informado, assumido como falso).
Changelog API iDClass - verso 101 (ou 257 decimal):
  • API get_afd retorna o AFD em formato da Portaria 671 caso informado mode=671;
  • API get_system_date_time retorna o "timezone" (fuso horrio) caso informado mode=671;
  • API set_system_date_time permite definir o "timezone" (fuso horrio) caso informado mode=671;
  • (Uso opcional) API add_users recebe o campo "cpf" em vez de "pis" caso informado mode=671;
  • (Uso opcional) API update_users recebe os campos "cpf" e "new_cpf" caso informado mode=671;
  • (Uso opcional) API load_users retorna usurios com "cpf" em vez de "pis" caso informado mode=671;
  • (Uso opcional) API template_merge recebe o campo "user_cpf" em vez de "user_pis" caso informado mode=671.
